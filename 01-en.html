SimplePress: Anatomy of a Static Site Generator

A Thirty-Year Journey on the Web

Before diving into the technical details, let me tell you a personal story. A story I probably share with many of you.

Thirty years ago, when I first started navigating the web, sites were static by definition. Hand-written HTML files, maybe with some animated GIF images and a questionable tiled background. It was a pioneering era: you learned HTML from scattered tutorials online, uploaded files via FTP, and each page was a small handcrafted artifact. There was something genuine in that simplicity.

Then WordPress came along, and with it the promise of making everything easier. And for a while, it was true. I enthusiastically embraced the world of dynamic CMS: databases, PHP, themes, plugins. You could do anything! Need a contact form? There's a plugin for that. A carousel? A plugin. SEO? Plugin. Every problem had its pre-packaged solution.

But complexity grows insidiously. One plugin leads to another, themes become frameworks, updates pile up, and before you know it you're managing a fragile ecosystem where every update risks breaking something. I spent more time debugging plugin conflicts than writing content. Maintenance had become a job... a job I was already doing eight hours a day.

There was a period of rejection, I admit. I stopped writing on the web partly because managing the whole setup had become a hassle. The site sat there, with its obsolete plugins and potential vulnerabilities, like an abandoned garden that makes you feel guilty every time you walk past it.

Then I decided to start again, but with a different approach. Minimalist. Only what I need. I wanted to return to that original simplicity, but with modern tools. No database, no heavy CMS, no dependencies that break. Just text files that become web pages. And that's how SimplePress was born.

The Advantages of Static Sites

A static site consists only of HTML, CSS, and JavaScript files. There's no database, no application server, no queries to optimize. The advantages are significant:

Speed: without server-side processing, pages are served instantly. Response time is limited only by network latency.

Security: no database means no SQL injection. No CMS means no vulnerabilities to patch. The attack surface is reduced to a minimum.

Costs: a static site can be hosted for free on GitHub Pages, Netlify, or a simple S3 bucket. No server costs, no scaling to manage.

Reliability: fewer components means fewer things that can break. A static site on a CDN has virtually guaranteed uptime.

Owning Your Content

There's another aspect that's particularly close to my heart, and it goes beyond technical considerations: ownership of your content.

We live in an era where we entrust our words, our thoughts, our creations to third-party platforms. Social networks, blogging services, newsletter platforms. It's convenient, sure. Someone else takes care of the infrastructure, you just have to write.

But at what cost? Your content exists as long as the platform decides it can exist. A policy change, an algorithm that decides you're less visible, a service that shuts down (remember Google+? Vine? Tumblr in its original form?), and years of effort can vanish or become inaccessible.

And then there's the issue of monetization of your content by others. When you write on a free platform, you are the product. Your texts, your interactions, your data feed advertising algorithms and business models that don't belong to you.

A static site, hosted on a service you control (even just a GitHub repository), gives you back this ownership. The files are yours, in an open and portable format. You can move them wherever you want, whenever you want. You can back them up, archive them, ensure they will exist as long as you decide.

That's why I believe a simple approach to web publishing is not just a technical choice, but also a principled one. A way to reclaim a piece of digital sovereignty in a world where we tend to delegate more and more.

For a personal blog, a portfolio, or a documentation site, these characteristics often outweigh the flexibility of a traditional CMS. And it's exactly in this context that SimplePress was born.

The Architecture of SimplePress

SimplePress follows a modular and clean architecture, organized into components with well-defined responsibilities. Let's see how the project is structured:

simplepress/
├── md-src/              # Markdown sources
│   ├── 01.md           # Numbered posts (chronological order)
│   ├── 02.md           # Other posts...
│   ├── 01-en.md        # English translations
│   └── chi-sono.md     # "About me" page
├── src/                # TypeScript code
│   ├── index.ts        # CLI entry point
│   ├── site-generator.ts
│   ├── markdown-parser.ts
│   ├── template-engine.ts
│   ├── types.ts
│   └── config.ts
├── build/              # Generated HTML output
└── dist/               # Compiled TypeScript

The separation is clear: content resides in md-src/ as Markdown files, source code lives in src/, and the final output is generated in build/. This approach keeps each component isolated and easily replaceable.

The TypeScript Modules

The heart of SimplePress consists of six TypeScript modules, each with a specific responsibility:

types.ts - The Foundations of the Type System

export interface Post {
  id: string;
  title: string;
  titleEn?: string;
  content: string;
  contentEn?: string;
  excerpt: string;
  excerptEn?: string;
  filename: string;
  order: number;
  createdAt: Date;
}

export interface Page {
  id: string;
  title: string;
  titleEn?: string;
  content: string;
  contentEn?: string;
  filename: string;
}

export type Language = 'it' | 'en';

These interfaces define the contract that runs through the entire application. A Post has a title, content, an excerpt (summary), and optionally the English versions of each. The order field determines chronological ordering: lower numbers represent older posts.

The Generation Flow

The generation process follows a linear and predictable flow. When we run npm run build, the following happens:

Phase 1: Content Discovery

The SiteGenerator starts by exploring the md-src/ directory:

private getPostFiles(): string[] {
  return fs.readdirSync(this.srcDir)
    .filter(file => {
      const isMarkdown = file.endsWith('.md');
      const isNumbered = /^\d+/.test(file);
      const isNotAboutPage = !file.startsWith('chi-sono');
      const isNotEnglishTranslation = !file.endsWith('-en.md');
      return isMarkdown && isNumbered && isNotAboutPage && isNotEnglishTranslation;
    })
    .map(file => path.join(this.srcDir, file));
}

Files are filtered with precise criteria: only Markdown files, with names starting with a number (for posts), excluding the "About me" page and English translations (which will be associated with their respective Italian posts).

The naming convention is elegant in its simplicity: 01.md is the first post, 01-en.md is its English translation. The system automatically recognizes this relationship and builds bilingual posts.

Phase 2: Markdown Parsing

The MarkdownParser handles transforming Markdown files into usable data structures. It uses the marked library for HTML conversion, but adds important features.

Metadata Extraction (Frontmatter)

SimplePress supports YAML frontmatter for specifying optional metadata:

---
title: Custom Title
titleEn: Custom English Title
---

# The Title in the Content
Here begins the actual content...

The parser extracts this metadata and separates it from the main content:

Handling Untranslated Posts

What happens if a user tries to access the English version of an untranslated post? SimplePress generates an "unavailable" page with an appropriate message, instead of showing an error or the Italian content. It's a touch of attention to user experience.

GDPR Compliance: The Cookie Banner

In an era of growing attention to privacy, SimplePress includes a cookie management system compliant with GDPR and the ePrivacy directive.

The Paradox of the Cookie-Free Site

Here's an interesting aspect: SimplePress doesn't use tracking or profiling cookies. Technically, no banner might be needed. However, regulations require informing users even about the use of technical cookies (such as those used to store consent itself).

The Implementation

The cookie banner appears on first access and offers two options: Accept or Reject.

function checkCookieConsent() {
  const consent = localStorage.getItem('cookieConsent');
  if (consent === 'rejected') {
    document.body.innerHTML = `
      <div class="cookie-rejected">
        <h2>You must accept cookies to access the site.</h2>
        <button onclick="resetCookieConsent()">I Accept</button>
      </div>
    `;
    return;
  }
  if (!consent) {
    document.getElementById('cookie-banner').style.display = 'block';
  }
}

If the user rejects cookies, access to the site is blocked. It may seem drastic, but it's a legitimate choice: the site transparently declares that it doesn't use tracking cookies, and the user can choose not to proceed.

Consent is stored in localStorage, which is technically not a cookie but local browser storage. This choice avoids the paradox of the "cookie to remember you don't want cookies."

The Build System

SimplePress offers several commands to manage the development cycle:

npm run setup     # Install dependencies and compile
npm run build     # Compile TypeScript and generate the site
npm run generate  # Generation only (TypeScript already compiled)
npm run watch     # Automatically regenerate on changes
npm run serve     # Local server for preview
npm run clean     # Remove generated files

Watch Mode

Watch mode is particularly useful while writing:

watch(): void {
  console.log(`Watching ${this.srcDir} for changes...`);
  
  fs.watch(this.srcDir, { recursive: true }, (eventType, filename) => {
    if (filename && (filename.endsWith('.md') || filename.includes('img/'))) {
      console.log(`File changed: ${filename}`);
      console.log('Regenerating site...');
      this.generate();
    }
  });
}

Every change to a Markdown file or image triggers a complete site regeneration. For a blog with a few dozen posts, this operation is instantaneous.

Image Handling

SimplePress automatically copies images from md-src/img/ to build/img/:

private copyImages(): void {
  const srcImgDir = path.join(this.srcDir, 'img');
  const buildImgDir = path.join(this.buildDir, 'img');

  if (!fs.existsSync(srcImgDir)) {
    return;
  }

  if (!fs.existsSync(buildImgDir)) {
    fs.mkdirSync(buildImgDir, { recursive: true });
  }

  const files = fs.readdirSync(srcImgDir);
  for (const file of files) {
    const srcFile = path.join(srcImgDir, file);
    const buildFile = path.join(buildImgDir, file);
    
    if (fs.statSync(srcFile).isFile()) {
      fs.copyFileSync(srcFile, buildFile);
    }
  }
}

In Markdown, images can be referenced with relative paths: ![Description](img/my-image.png).

Extensibility and Customization

SimplePress is designed to be easily customizable:

Configuration: by modifying src/config.ts you can change the title, description, author, color palette, and all translations.

Styles: the CSS in template-engine.ts can be modified to completely change the site's appearance.

Structure: by adding new page types in types.ts and related generation methods, you can extend the system.

Translations: by adding new keys in translations, support for other languages is immediate (although it would require changes to the Language type).

Conclusions

SimplePress represents a minimalist but complete approach to static site generation. It doesn't compete with Hugo or Jekyll for features, but offers something different: a readable and modifiable codebase that can be fully understood in an afternoon.

For those managing a personal blog or a small site, this level of control is valuable. There are no mysterious dependencies, no magical behaviors to debug, no updates that risk breaking everything.

The choice of TypeScript ensures type safety and autocompletion during development. The modular architecture allows modifying one component without impacting others. Native bilingual support avoids the complexity of external plugins. And out-of-the-box GDPR compliance eliminates a common concern for European sites.

Want to try SimplePress? The setup is simple:

git clone [repository]
cd simplepress
npm run setup
# Add your content in md-src/
npm run build
npm run serve
# Visit http://localhost:8000

Your static, bilingual, and GDPR-compliant blog is ready in minutes.
